# Production nginx config — tracked copy

`cyprusvipestates.conf` is a **read-only mirror** of the live file at
`/etc/nginx/sites-enabled/cyprusvipestates` on the production VPS
(`root@72.60.89.239`). It is not deployed from here automatically — nginx
config changes are made by hand on the VPS, then the updated file is copied
back into this directory so it's never an invisible, untracked layer again
(see the incident this fixes, below).

## Why this exists

2026-07-19: an EN-migration redirect fix was built and deployed entirely at
the Next.js `middleware.ts` level, tested and confirmed correct in local dev
— then found to have **no effect in production**, because nginx already has
its own blanket `/en/` → root rewrite (`location ^~ /en/`) that intercepts
every request before it ever reaches the app. That rule predates this
incident, lived only on the VPS, and appears in no repo, doc, or commit
anywhere — so nobody researching or changing `/en/` routing behavior had any
way to know it existed short of SSHing in and reading `/etc/nginx/`. This
directory + the note in `DEPLOYMENT.md` exist so that never happens again.

## Keeping this in sync

There is currently no automation enforcing that this file matches the live
one — if you edit nginx directly on the VPS, **copy the file back here in
the same session**:

```bash
scp -i ~/.ssh/cvp_vps root@72.60.89.239:/etc/nginx/sites-enabled/cyprusvipestates ops/nginx/cyprusvipestates.conf
git add ops/nginx/cyprusvipestates.conf
git commit -m "ops/nginx: sync after <describe the live change>"
```

Also take a timestamped on-VPS backup before any edit (independent of git,
in case you need to roll back without a deploy):

```bash
ssh -i ~/.ssh/cvp_vps root@72.60.89.239 "mkdir -p /root/nginx-backups && cp /etc/nginx/sites-enabled/cyprusvipestates /root/nginx-backups/cyprusvipestates.conf.\$(date -u +%Y%m%d_%H%M%S)"
```

Always `nginx -t` on the VPS before reloading, and `systemctl reload nginx`
(never `restart`) to avoid dropping in-flight connections.

## What's NOT tracked here (yet)

- `design.cyprusvipestates.com` (staging vhost) and `private-collection` —
  same "untracked until someone copies it over" problem, not addressed by
  this batch. Worth doing the same treatment for staging's config if/when
  it's next touched.
- The Certbot-managed SSL blocks (`ssl_certificate`, etc.) in this file are
  regenerated by `certbot renew` — expect diffs there that don't reflect a
  deliberate routing change.
